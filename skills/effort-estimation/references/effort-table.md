# 研发工时基准表（人日 pd）

本表为"工作项类型 × 复杂度"的基准工时（单位：人日 pd），用于 `effort-estimation` skill 查表计算。
用户可直接修改本表，无需改动 SKILL.md。

## 基准工时表（人日 pd）

| 工作项类型 | 简单 | 中等 | 复杂 |
| ---------- | ---- | ---- | ---- |
| 需求       | 0.5  | 1.0  | 2.0  |
| 设计       | 0.5  | 1.0  | 2.0  |
| 前端       | 1.0  | 2.0  | 4.0  |
| 后端       | 1.0  | 2.5  | 5.0  |
| 测试       | 0.5  | 1.5  | 3.0  |
| 运维       | 0.5  | 1.0  | 2.0  |

> 设计依据：基础值参考 COCOMO II 阶段占比（需求 10-15% / 设计 5-10% / 前端 15-25% / 后端 25-35% / 测试 15-20% / 运维 5-10%）与功能点法复杂度分级回推。后端 / 前端在中大型业务中承担主要编码工作量，因此复杂级基准较高。

## 复杂度判定准则

按以下四个维度综合判断，每个维度 0-3 分，总分映射到复杂度：

| 维度          | 说明                                         |
| ------------- | -------------------------------------------- |
| 数据实体数    | 涉及的领域模型 / 数据库表数量                |
| 交互点数      | UI 交互节点 / API 端点数量                   |
| 算法/规则密度 | 业务规则数量、是否涉及复杂校验、计算、状态机 |
| 集成点数      | 是否对接第三方系统、消息队列、外部服务       |

- **简单**（总分 0-3）：单实体 CRUD、无复杂规则、无外部集成
- **中等**（总分 4-7）：2-3 个实体联动、含业务规则、可能有 1 个集成点
- **复杂**（总分 8-12）：多实体、复杂规则/算法、多个集成点、状态机或事务

## 团队构成调整因子

加权生产率系数 `k` 按团队角色分布计算：

```
k = (senior × 0.8 + middle × 1.0 + junior × 1.3) / total
```

最终工时 = 基准工时 × k（四舍五入到 0.5 pd 粒度，保留 1 位小数）

### 角色系数

| 角色   | 系数 | 说明                               |
| ------ | ---- | ---------------------------------- |
| senior | 0.8  | 资深研发，产出效率高于基准         |
| middle | 1.0  | 中级研发，与基准工时一致           |
| junior | 1.3  | 初级研发，需要更多时间完成同样工作 |

### 预设团队构成

| 预设   | senior | middle | junior | 计算系数 k |
| ------ | ------ | ------ | ------ | ---------- |
| 偏资深 | 2      | 1      | 0      | 0.87       |
| 均衡   | 1      | 2      | 1      | 1.03       |
| 偏初级 | 0      | 1      | 2      | 1.20       |

### 示例

基准 2.0 pd 的后端-中等任务，团队为 `senior=1,middle=2,junior=1`：

- k = (1×0.8 + 2×1.0 + 1×1.3) / 4 = 4.1 / 4 = 1.025
- 最终工时 = 2.0 × 1.025 = 2.05 → 四舍五入到 0.5 粒度 = **2.0 pd**

## 非功能要求处理规则

非功能要求（NFR：性能、可用性、合规等）**不另起第 7 类模块**，而是映射到现有六类，避免扰乱输出结构：

| NFR 类型                | 示例          | 处理方式                                                             |
| ----------------------- | ------------- | -------------------------------------------------------------------- |
| 性能（TPS / RT / 并发） | 下单 200 TPS  | 把受影响功能点的 **后端** 复杂度提升一档（中等 → 复杂）              |
| 可用性 / 灾备           | 99.9% SLA     | 在最相关业务模块下新增 1 项 **运维-复杂**（限流、多活、监控）        |
| 数据保留 / 审计 / 合规  | 订单保留 3 年 | 新增 1 项 **后端-中等**（归档/冷热分层）+ 1 项 **运维-中等**（策略） |
| 安全（加密、数据脱敏）  | 手机号脱敏    | 抬升相关 **后端** 复杂度；如无现有接入点，新增 **后端-中等**         |

## 人工复核触发条件

- 单工作项工时 > 5 pd → 提示用户考虑拆细
- 单业务模块总工时 > 20 pd → 建议二次拆分模块
- 复杂度判定落在边界（如简单/中等之间分数为 3 或 4）→ 在输出中标注 "⚠ 边界"
