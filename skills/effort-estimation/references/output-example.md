## 1. 拆解依据（示例）

> 业务模块按需求文档二级标题划分（含订单 / 支付 / 物流 3 个模块）；复杂度按新粒度判定（前端=页面、后端=业务动作）；团队系数 k = 1.03（均衡）。

## 4. 工作内容填写约定（关键）

| 类型           | ✓ 这样写                                                  | ✗ 不要这样写                       |
| -------------- | --------------------------------------------------------- | ---------------------------------- |
| 前端           | `下单页` / `订单列表页` / `个人中心`                      | `Vue 组件` / `React Hook` / `路由` |
| 后端           | `新增订单` / `查询订单列表` / `订单状态校验` / `库存扣减` | `POST /orders` / `/api/orders/:id` |
| 运维           | `部署流水线` / `监控告警` / `日志系统`                    | `写 Dockerfile` / `改 YAML`        |
| 需求/设计/测试 | `-` （模块级）                                            | 不要在这里细分子任务               |

## 5. 反例对照（相对于 v2 的改进）

| 场景           | v2 做法          | v3 做法                                           |
| -------------- | ---------------- | ------------------------------------------------- |
| 订单-下单 后端 | 1 行 中等 2.5 pd | 5 行（新增/查询/修改/取消/库存逻辑），合计 2.4 pd |
| 订单-下单 前端 | 1 行 中等 2.0 pd | 2 行（下单页 + 购物车页），合计 2.0 pd            |
| 需求 / 设计    | 1 行 中等 1.0 pd | 1 行 中等 **2.0 pd**（翻倍，反映真实投入）        |
| 运维           | 1 行 中等 1.0 pd | 4 行（部署 / 监控 / 日志 / 应急），合计约 2.5 pd  |

> v3 的总量比 v2 略高，这是刻意为之 —— v2 低估了需求/设计/运维的实际投入。
